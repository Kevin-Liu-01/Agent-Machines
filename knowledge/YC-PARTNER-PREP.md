## What is it? (no buzzwords)

[↑ All questions](#question-index)

**Say this:**

It's a website where you deploy a worker — a computer in the cloud that runs an AI agent that's actually set up to do real work. Not a chat window. Not an empty server you have to configure yourself. You pick what kind of worker you want — coding, research, ops — and we spin up a machine that already has the agent installed, a hundred and sixty-one pre-loaded skills, connections to tools like GitHub and Stripe and your email, a schedule for overnight jobs, and a dashboard where you can see everything it did. It remembers state between sessions because it has a real disk. You don't have to keep your laptop open. ChatGPT gave you a chat box in seconds; we give you a worker that finishes tasks in seconds.

## How does it work? (no buzzwords)

[↑ All questions](#question-index)

**Say this:**

You sign in, pick a worker type — code, research, ops — connect whatever it needs through normal sign-in buttons, GitHub or Gmail, not editing config files, and click deploy. In about thirty seconds there's a computer running in the cloud. You just get a worker that's ready to work.

We're a cloud worker platform where you can run four runtimes across three substrates, swap models through two hundred plus gateway routes, supervise a fleet from one dashboard, and accumulate procedures as SKILL.md files on disk that compound every session — not trapped in chat history. What makes us different from E2B is we're not selling the box, we're selling the worker that's already inside it, configured and observable. What makes us different from internal agent teams at big companies is they're building bespoke harnesses for delegation inside old org charts; we're productizing the harness itself for mass distribution, composable from the open skill and MCP ecosystem.

The observation layer alone is a different product category. Infra vendors give you SSH and logs. We give you chat replay, session timelines, tool traces, skill invocations, artifact browser, usage and cost — Datadog-shaped stickiness for agents. You can't orchestrate what you can't see, and you can't trust a worker you can't audit. That's why this isn't a feature of Dedalus or E2B — they optimize boot time and pricing per CPU. We optimize agent completion, harness composition, and distribution. Neutrality across substrates is the whole point.

## Technical moat (deep script)

[↑ All questions](#question-index)

**Use when they ask:** *"What's defensible?"* / *"What's actually hard here?"* / *"Why won't E2B add this?"*

**Anchor:** Seven layers of compounding — bootstrap, gateway, routing, harness, skills, observation, programmatic surface.

**Say this:**

I'll walk you through what's actually built, because this is not a landing page on top of someone else's API.

**First, the deploy pipeline.** When you click deploy, we provision through a `MachineProvider` abstraction — Dedalus SDK, E2B SDK, Sprites SDK — each with different provision, exec, sleep, wake, and destroy semantics. Then we run a multi-phase bootstrap that's idempotent and resumable: system dependencies, UV and Node toolchains, clone and install the agent runtime, seed a hundred and sixty-one skills from our registry, install git reload hooks, optional Cursor bridge, configure Hermes or OpenClaw with model routing, register MCP servers from our catalog, seed cron jobs, start the gateway with substrate-specific lifecycle — sandbox gateways get a different path than persistent microVMs — and install closed-loop CLIs for Vercel, Stripe, GitHub, and the rest. If bootstrap fails mid-phase, we resume from the last completed phase, not from zero. If the machine sleeps and wakes, we detect missing artifacts and re-bootstrap in the background. Even Dedalus, which benchmarks best on boot latency in our harness, still has a massive MDX setup doc with SSH and config surgery if you do it yourself. We collapse that to clicks.

**Second, the gateway.** This is the reachability and security layer between the user's browser and an agent running on arbitrary infra. We generate per-machine API keys, configure upstream model routing — direct provider keys or Vercel AI Gateway — stream chat and tool calls back through our dashboard, and handle computer use, artifacts, and browser automation paths. Gateway auth is off the model path. The agent writes to disk on the machine; the dashboard reads observation data through our APIs. That's a different trust boundary than "SSH into a box and run curl."

**Third, triple routing.** Runtime router — Hermes, OpenClaw, Claude Code, Codex. Substrate router — E2B, Sprites, Dedalus. Model router — two hundred plus OpenAI-compatible endpoints. OpenRouter proved neutral routing is a durable business. We're doing it for agents and containers, not just tokens.

**Fourth, the harness registry.** A hundred and sixty-one SKILL.md procedures synced from our knowledge base, thirty-five MCP catalog entries with ranked service routes — which MCP to use for which task, in what order — plus closed-loop CLIs and agent-native tools that vary by runtime. This isn't a static Docker image. It's a living registry that compounds every week. I've personally accumulated five hundred plus skills in my LLM wiki; the product ships the battle-tested subset. Users inherit that curation on deploy.

**Fifth, SKILL.md as protocol.** When an agent completes a complex task, it can save the procedure as a versioned skill on disk under `~/.agent-machines/`. Next session, next cron run, next worker — the procedure is there. Switching cost grows every session because the worker's capabilities are literally accumulated on the filesystem, not exportable to ChatGPT. That's intentional.

**Sixth, observation.** Sessions, logs, tool traces, skill invocations, artifacts, usage, cost — unified across heterogeneous runtimes. Activity heatmap, chat replay, fleet monitor. This is the Datadog layer for agents. Stickiness lives here. Once your workers run overnight cron and you audit them through our dashboard, you're not going back to SSH.

**Seventh, programmatic surface — roadmap but architected for now.** The dashboard is table stakes. The moat is MCP and CLI so a head agent can `am provision` a code reviewer and a deploy monitor, route work between them, observe results, tear down. Humans deploy today. Agents deploy tomorrow. Recursive platform growth.

**Why incumbents don't trivially copy:** OpenAI is model-first and vendor-locked. E2B is infra-first with no harness, presets, or agent observation UX. OpenClaw is runtime-first with no account, fleet, or distribution. Internal enterprise teams build closed stacks for themselves, not consumable deploy surfaces. We're the only live product that composes all layers neutrally.

**If they push "six months and someone copies you":** The harness compounds, skills accumulate on user machines, observation history sticks, and substrate abstraction means we improve every lane without user migration. ### Technical depth



**Q: What's actually built?**

---



**Q: Hardest technical problem?**

[↑ All questions](#question-index)

**Say this:** Same harness, reliably, on heterogeneous substrates — that's the core hard problem. E2B sandboxes are ephemeral and need a different gateway lifecycle than Dedalus persistent microVMs with sleep and wake. Sprites has its own API semantics. Bootstrap has thirteen phases — system deps through gateway start — and every phase has to be idempotent, resumable if it fails mid-way, and re-runnable on wake without clobbering skills accumulated on disk. On top of that, gateway auth and streaming have to work off the model path so the dashboard can show tool calls, artifacts, and computer use in real time across four different runtimes that don't speak the same protocol natively. Observation has to unify all of that into sessions and cost a human actually trusts. Making that feel like one click — when the underlying systems are three different vendor SDKs with different failure modes — that's the engineering. Everything else is composition on top of a control plane that actually works.

---



**Q: Why three substrates?**

[↑ All questions](#question-index)

**Say this:** OpenRouter pattern — no lock-in. Route to the best lane per workload. E2B for ephemeral sandbox tasks. Sprites for their microVM shape. Dedalus for persistent machines with fast boot and sleep — actually benchmarks best in our harness at around two hundred fifty milliseconds boot. User shouldn't have to become a substrate expert; they pick from a dropdown or we pick for them based on preset. If a vendor improves, we swap lanes without user migration because the harness is substrate-agnostic. Neutrality is a feature — E2B shouldn't only run E2B agents any more than OpenRouter should only route to OpenAI.
